# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository structure

This repo contains **two parallel, independently-runnable implementations of the same site** (Zahau Music School — an online music academy: piano, guitar, drums, vocals, theory). They share the same visual design, Supabase backend, and largely the same component/page inventory, but are built on different frameworks:

- **Root (`/`)** — TanStack Start (React 19 + TanStack Router + Vite + Nitro). Package name `tanstack_start_ts`.
- **`next-app/`** — Next.js 15 App Router, its own `package.json`, `node_modules`, and `.env.local`.

When making a change that affects site behavior (a form, a page, a data query, an auth flow), check whether the equivalent needs to change in **both** apps — e.g. `src/routes/courses.$slug.tsx` and `next-app/app/courses/[slug]/page.tsx` + `course-detail-client.tsx` are two implementations of the same course-detail page, and `src/lib/site.functions.ts` (TanStack server functions) mirrors `next-app/app/api/*/route.ts` (Next API routes) — both call the same Supabase tables.

Root-level scratch/one-off scripts (`fetch_live_data.cjs`, `parse_bundle.js`, `query_courses.cjs`, `scratch_update_db.js`/`.cjs`, `test_signin.cjs`, `test_claims.cjs`, `live_bundle.js`, `live_courses.json`, `live_site.html`) are ad hoc data/debugging scripts, not part of either app's build.

## Commands

Run from the repository root for the TanStack Start app:

```
npm run dev       # start Vite dev server
npm run build     # production build
npm run preview   # serve the production build locally
npm run lint      # ESLint
npm run format    # Prettier --write
```

Run from `next-app/` for the Next.js app:

```
cd next-app
npm run dev       # next dev
npm run build     # next build
npm run start     # serve production build
npm run lint      # next lint
npm run format    # Prettier --write
```

There is no automated test runner configured in either app. Verify changes with `lint` + `build`, then manually exercise the affected route.

Package manager: root uses `bun.lock` (Bun) per `bunfig.toml`, but `npm ci`/`npm run` also work off `package-lock.json`. `next-app/` has its own `package-lock.json` and is npm-only.

## Architecture

### TanStack Start app (root `src/`)

- **Routing**: file-based via `src/routes/`, conventions documented in `src/routes/README.md`. `src/routeTree.gen.ts` is auto-generated — never edit by hand. `src/routes/__root.tsx` is the single app shell (header/footer/toaster/query client); it hides the header/footer on `/dashboard`, `/book-demo`, and `/auth`.
- **Auth-gated routes** live under `src/routes/_authenticated/` — `dashboard.tsx` (student portal only; redirects admins to `/admin`) and `admin.tsx` (admin console; redirects non-admins to `/dashboard`). The layout route (`_authenticated/route.tsx`) calls `supabase.auth.getUser()` in `beforeLoad` and redirects to `/auth` if unauthenticated. It has `ssr: false` and special-cases the brief window right after a Google OAuth redirect (waits for Supabase to persist the session from the URL hash).
- **Supabase clients** in `src/integrations/supabase/` — several files there carry a `// This file is automatically generated. Do not edit it directly.` header (`client.ts`, `client.server.ts`, `auth-middleware.ts`, `types.ts`); treat those as generated and make schema/type changes via migrations + regeneration rather than hand-editing:
  - `client.ts` — browser client (publishable key), session persisted to `localStorage`.
  - `client.server.ts` (`supabaseAdmin`) — service-role client that bypasses RLS. Server-only; never import from client code.
  - `auth-middleware.ts` (`requireSupabaseAuth`) — Bearer-token verification middleware; per its own header comment it is currently unused.
- **Server functions**: `src/lib/site.functions.ts` defines all mutations/queries as TanStack `createServerFn` handlers (leads, demo bookings, courses, fees, lessons, posts, Razorpay order creation/verification, admin role assignment). Each handler lazy-imports `supabaseAdmin` and validates input with Zod. This is the single source of truth for backend logic in the TanStack app — the Next.js app's API routes duplicate this logic independently.
- **Shared checkout helpers**: `src/lib/razorpay.ts` (script loader, typed checkout constructor, `getRazorpayKeyId`, `INSTRUMENTS`, `instrumentToSlug` — the slug map must stay in sync with the seeded course slugs) and `src/lib/fee-packages.ts` (`FeePackage` type, `normalizeFeePackages`, static `FALLBACK_PACKAGES`) are used by `/fees`, `/book-demo`, and the student dashboard. Razorpay keys come only from env (`VITE_RAZORPAY_KEY_ID` client-side, `RAZORPAY_KEY_ID`/`RAZORPAY_KEY_SECRET` server-side) — never hardcode fallbacks.
- **Server-only config**: `src/lib/config.server.ts` documents the three env-access patterns in use — `.server.ts` module-scope helpers (wrap reads in a function; on edge runtimes env binds per-request, so module-scope `process.env` reads resolve to `undefined`), inline `process.env` inside a single `createServerFn` handler, and `import.meta.env.VITE_*` for values that are intentionally public (never put secrets behind the `VITE_` prefix).
- **Authorization**: role-based (`student`/`admin`) via a `user_roles` table and a Postgres `has_role` function (see `supabase/migrations/`); `RootComponent` in `__root.tsx` fetches the caller's role client-side after auth state changes to decide whether to expose admin UI.

### Next.js app (`next-app/`)

- App Router pages under `next-app/app/`, mirroring the TanStack routes (`courses/[slug]`, `dashboard`, `fees`, `book-demo`, `auth`, etc.). Server logic lives in `next-app/app/api/*/route.ts` handlers instead of TanStack server functions, backed by `next-app/lib/supabase/` and `next-app/utils/supabase/` clients (browser/server/middleware split, same publishable-key/service-role-key pattern as the root app).
- Client-side interactive pieces are split into `*-client.tsx` components (e.g. `course-detail-client.tsx`) alongside their server `page.tsx`, following Next.js server/client component conventions.

### Database (`supabase/`)

- `supabase/migrations/` holds timestamped SQL migrations — add schema changes as new migration files, don't edit past ones. `setup_supabase_db.sql` at the repo root is a consolidated setup script for the Supabase SQL editor.
- Supabase is accessed via the MCP server configured in `.mcp.json` (project ref `srxogdtbokmlfibjkvcw`) and the `supabase` / `supabase-postgres-best-practices` skills under `.agents/skills/`.
- Key tables referenced from application code: `courses`, `fees`, `lessons`, `posts`, `leads`, `newsletter_subscribers`, `enrollments`, `user_roles`.

### Environment variables

Root `.env` (server + Vite-prefixed public vars): `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_PROJECT_ID` and their `VITE_`-prefixed public counterparts, `GOOGLE_CLIENT_ID`/`SECRET` (OAuth), `SMTP_*` (transactional email), `RAZORPAY_KEY_ID`/`SECRET` and `VITE_RAZORPAY_KEY_ID` (payments). `next-app/.env.local` uses the Next.js convention: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`.

## Coding conventions

- TypeScript, React function components. Prettier is authoritative (100-char lines, semicolons, double quotes, trailing commas) — run `npm run format` before committing.
- Component exports: PascalCase (`BookDemoModal`). Ordinary component filenames: lowercase-hyphenated (`book-demo-modal.tsx`).
- TanStack route files follow the router's file-based convention (bare `$` for dynamic segments, e.g. `courses.$slug.tsx`; `_authenticated` for a pathless layout) — see `src/routes/README.md` for the full table.
